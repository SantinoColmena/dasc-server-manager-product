# Decisión - GitHub interno y soporte real para clientes

## Contexto

Durante la Fase 6 se ha empezado a preparar el sistema de soporte y tickets de DASC Server Manager.

Se detecta una diferencia importante:

- GitHub Issues puede servir al equipo para organizar incidencias internas.
- Una PyME real no debería usar GitHub para pedir soporte.

## Decisión

GitHub Issues queda definido únicamente como herramienta interna del proyecto.

No se ofrecerá GitHub como canal directo de soporte para clientes.

## Motivo

Una pequeña empresa normalmente no quiere ni debe:

- Crear cuenta de GitHub.
- Entrar al repositorio.
- Aprender a crear issues.
- Ver documentación técnica interna.
- Exponer información sensible en un repositorio.
- Gestionar incidencias como si fuera un equipo técnico.

DASC debe vender simplicidad, no trasladar complejidad al cliente.

## Enfoque correcto

El cliente usará canales sencillos:

- Email de soporte.
- Formulario web.
- WhatsApp o teléfono en planes que lo incluyan.
- Panel de ayuda dentro de DASC en una fase futura.
- Portal de cliente en una fase posterior si el producto crece.

El equipo de DASC será quien registre internamente la incidencia en GitHub Issues o en la herramienta que se use en ese momento.

## Flujo correcto de soporte

1. El cliente comunica el problema por un canal sencillo.
2. El equipo revisa si falta información.
3. El equipo crea un ticket interno.
4. El ticket se clasifica por prioridad y módulo.
5. Se investiga.
6. Se responde al cliente por su canal original.
7. Se documenta la solución internamente.
8. Se cierra el ticket.

## Uso de GitHub Issues

GitHub Issues puede usarse para:

- Errores internos.
- Mejoras pendientes.
- Validaciones técnicas.
- Incidencias detectadas en pruebas.
- Seguimiento del desarrollo.
- Registro interno de soporte recibido por otros canales.

GitHub Issues no debe presentarse como:

- Portal de cliente.
- Mesa de ayuda profesional.
- Sistema comercial de tickets.
- Canal obligatorio para PYMES.
- Sustituto de un soporte humano claro.

## Impacto en Fase 6

La Fase 6 continúa siguiendo el tracker, pero con esta corrección:

- R-048 queda en curso hasta que el producto esté más pulido.
- R-049 queda como base interna provisional, no como ticketing comercial.
- R-050 puede avanzar como informe mensual interno/cliente, siempre que sea simple y útil.
- Las ventas se dejan para el final, cuando el producto sea presentable y no solo documentado.

## Conclusión

DASC Server Manager debe mantener GitHub como herramienta interna del equipo.

El cliente final debe ver una experiencia sencilla, guiada y cercana.
